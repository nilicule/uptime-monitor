# uptime-monitor

A Cloudflare Workers project that monitors HTTP endpoints and TCP ports on a 5-minute cron schedule. Results are stored in KV and exposed via a pre-built dashboard snapshot.

## Project structure

```
uptime-monitor/
├── wrangler.toml.example  # Template committed to repo (placeholder KV IDs)
├── wrangler.toml          # Generated by bootstrap.sh — gitignored, never committed
├── bootstrap.sh           # One-time setup: creates KV namespaces + sets secret
├── .gitignore
├── scripts/
│   └── monitors.sh        # Manage monitors (list / add / remove)
└── src/
    ├── worker.js          # Scheduled handler, fetch handler, KV writes
    └── checks.js          # HTTP and TCP check implementations
```

`wrangler.toml` is intentionally gitignored. It is generated from `wrangler.toml.example` by `bootstrap.sh` with your real KV namespace IDs spliced in. This keeps infrastructure identifiers out of the repo.

## Setup

### Prerequisites

- [Wrangler CLI](https://developers.cloudflare.com/workers/wrangler/) installed and logged in
- `jq` installed (used by `scripts/monitors.sh`)

### Bootstrap

```bash
./bootstrap.sh
```

This will:
1. Create the `UPTIME_KV` KV namespace (production + preview)
2. Generate `wrangler.toml` from `wrangler.toml.example` with the real KV IDs injected automatically
3. Prompt you for your monitors JSON and set it as the `MONITORS_CONFIG` secret

`wrangler.toml` is gitignored. Run `bootstrap.sh` once per clone — it is safe to re-run if you need to recreate it.

### Monitor configuration

Monitors are stored as a Cloudflare Worker secret named `MONITORS_CONFIG` (a JSON array). The secret is never committed to the repo. A readable mirror is written to `config:monitors` in KV on each cold start so `scripts/monitors.sh` can read it back.

```json
[
  { "id": "plex", "name": "Plex", "type": "http", "url": "https://plex.example.com", "expectedStatus": [200, 401] },
  { "id": "nas",  "name": "NAS",  "type": "tcp",  "host": "nas.example.com", "port": 5000 }
]
```

## Check types

| Type | Method | Timeout | What's measured |
|------|--------|---------|-----------------|
| `http` | HEAD, follows redirects | 10 s | Status code, response time, final URL |
| `tcp` | `cloudflare:sockets` connect | 5 s | TCP connect latency |

## KV data model

| Key | TTL | Purpose |
|-----|-----|---------|
| `latest:{id}` | 1 hour | Most recent result per monitor |
| `result:{id}:{ts}` | 90 days | Raw per-check result |
| `summary:{id}:{YYYY-MM-DD-HH}` | 90 days | Hourly aggregate bucket |
| `dashboard:snapshot` | none | Pre-built payload for the status page |
| `config:monitors` | none | Readable mirror of `MONITORS_CONFIG` secret |

## Deploy

```bash
wrangler deploy
```

The worker runs automatically every 5 minutes via cron (`*/5 * * * *`).

## Testing locally

### 1. Create `.dev.vars`

Secrets set via `wrangler secret put` are only available in deployed workers. For local dev, create a `.dev.vars` file (gitignored) in the project root:

```bash
# .dev.vars
MONITORS_CONFIG=[{"id":"plex","name":"Plex","type":"http","url":"https://plex.example.com","expectedStatus":[200,401]}]
```

### 2. Start the dev server

`--test-scheduled` is required to enable the `/__scheduled` trigger endpoint:

```bash
wrangler dev --test-scheduled
```

### 3. Trigger a check round

In a second terminal:

```bash
curl "localhost:8787/__scheduled?cron=*%2F5+*+*+*+*"
# → Ran scheduled event
```

### 4. Verify results

Use the API endpoints while the dev server is running — `wrangler kv key list` reads **remote** KV and won't show local dev results:

```bash
curl localhost:8787/api/snapshot
curl localhost:8787/api/monitor/plex
```

## Verifying KV results

```bash
# List all keys
wrangler kv key list --binding=UPTIME_KV --remote

# Latest result for a monitor
wrangler kv key get --binding=UPTIME_KV --remote "latest:plex"

# Dashboard snapshot
wrangler kv key get --binding=UPTIME_KV --remote "dashboard:snapshot"
```

## Live logs

```bash
wrangler tail
```

## Managing monitors

```bash
./scripts/monitors.sh list     # Pretty-print current monitors from KV
./scripts/monitors.sh add      # Interactively add a monitor
./scripts/monitors.sh remove   # Interactively remove a monitor
```

`add` and `remove` both update `config:monitors` in KV and re-set the `MONITORS_CONFIG` secret.

## HTTP API

| Endpoint | Description |
|----------|-------------|
| `GET /healthz` | Returns `200 OK` |
| `GET /api/snapshot` | Returns the full `dashboard:snapshot` payload |
| `GET /api/monitor/:id` | Returns raw check results for the last 24 hours |
