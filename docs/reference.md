# Technical Reference

## Project structure

```
uptime-monitor/
├── wrangler.toml.example  # Template committed to repo (placeholder KV IDs)
├── wrangler.toml          # Generated by bootstrap.sh — gitignored, never committed
├── bootstrap.sh           # One-time setup: creates KV namespaces + sets secret
├── scripts/
│   └── monitors.sh        # Manage monitors (list / add / remove)
└── src/
    ├── worker.js          # Scheduled handler, fetch handler, KV writes
    ├── checks.js          # HTTP and TCP check implementations
    └── page.js            # Status page HTML (served at GET /)
```

`wrangler.toml` is intentionally gitignored. It is generated from `wrangler.toml.example` by `bootstrap.sh` with your real KV namespace IDs spliced in.

## Check types

| Type | Method | Timeout | What's measured |
|------|--------|---------|-----------------|
| `http` | HEAD (falls back to GET on 405), follows redirects | 10 s | Status code, response time, final URL |
| `tcp` | `cloudflare:sockets` connect | 5 s | TCP connect latency |

## KV data model

| Key | TTL | Purpose |
|-----|-----|---------|
| `latest:{id}` | 1 hour | Most recent result per monitor |
| `result:{id}:{ts}` | 90 days | Raw per-check result |
| `summary:{id}:{YYYY-MM-DD-HH}` | 90 days | Hourly aggregate bucket |
| `dashboard:snapshot` | none | Pre-built payload for the status page |
| `config:monitors` | none | Readable mirror of `MONITORS_CONFIG` secret |

`dashboard:snapshot` is rebuilt on every cron run. The status page reads only this one key per page load regardless of how many monitors are configured.

## HTTP API

| Endpoint | Description |
|----------|-------------|
| `GET /` | Status page |
| `GET /healthz` | Returns `200 OK` |
| `GET /api/snapshot` | Returns the full `dashboard:snapshot` payload |
| `GET /api/monitor/:id` | Returns raw check results for the last 24 hours |

## Monitor configuration

Monitors are stored as a Cloudflare Worker secret named `MONITORS_CONFIG` (a JSON array). The secret is never committed to the repo. A readable mirror is written to `config:monitors` in KV on each cold start.

```json
[
  { "id": "plex", "name": "Plex", "type": "http", "url": "https://plex.example.com", "expectedStatus": [200, 401] },
  { "id": "nas",  "name": "NAS",  "type": "tcp",  "host": "nas.example.com", "port": 5000 }
]
```

## Local development

### 1. Create `.dev.vars`

Secrets set via `wrangler secret put` are only available in deployed workers. For local dev, create a `.dev.vars` file (gitignored):

```
MONITORS_CONFIG=[{"id":"plex","name":"Plex","type":"http","url":"https://plex.example.com","expectedStatus":[200,401]}]
```

### 2. Start the dev server

`--test-scheduled` enables the `/__scheduled` trigger endpoint:

```bash
wrangler dev --test-scheduled
```

### 3. Trigger a check round

In a second terminal:

```bash
curl "localhost:8787/__scheduled?cron=*%2F5+*+*+*+*"
```

### 4. Verify results

Use the API while the dev server is running — `wrangler kv key list` reads remote KV and won't show local dev results:

```bash
curl localhost:8787/api/snapshot
curl localhost:8787/api/monitor/plex
```

## Verifying remote KV

```bash
wrangler kv key list --binding=UPTIME_KV --remote
wrangler kv key get --binding=UPTIME_KV --remote "latest:plex"
wrangler kv key get --binding=UPTIME_KV --remote "dashboard:snapshot"
```
